02 · Summary

기본 요청사항

목표 · 역할 분담 · 요구 기능 · 예약 · CMS 범위 · 데이터 제공 · 확인 필요 사항 · 넥스트 액션을 한 장으로 요약했습니다. 각 요구를 클릭하면 03 요구사항 정리의 상세로 이동합니다.

01프로젝트 목표

모든 요구가 향하는 공통 방향
  • 운영 독립성 — 콘텐츠·예약·다국어·권한을 운영자가 개발 없이 직접 관리.
  • 고객 여정·데이터 주도권 — 탐색→상담·예약→마이페이지를 홈페이지 안에서 완결, 데이터는 당사 축적.
  • 반응형 글로벌 플랫폼 — 단일 URL 반응형 · 경로 기반 다국어 · 콘텐츠·표현 분리로 확장.
  • 측정 기반 — 검색 노출 · 유입·전환 측정을 오픈 시점부터 (GEO는 선택 견적).

누가 무엇을 만드나 — 세부 판단의 단일 기준 · 디자인 협력사 산출 범위는 잠정(08 확정)
주체범위
듀당스브랜드 컨셉·로고·화면 디자인 · HTML/CSS 퍼블리싱은 참조용 잠정
시그퍼블리싱을 참조해 Next.js 컴포넌트로 재구축 · 인터랙션·라우팅·API 연동·폼·CMS 전부
비쥬웍스백엔드 API·DB · 데이터 이관 · 인증/CRM 연동 · GA 계정 · 인프라
  • 결과물 — PC/모바일 통합 반응형 웹(단일 URL), 모바일은 전용 레이아웃 별도 디자인.
  • 다국어 — 시그 구축 범위는 KO + EN. 중국어·일본어·베트남어 번역·운영은 비쥬웍스.

04예약 · 인증

미팅 확정 사항
  • 예약 시스템 — 현 마루앱 관리자를 대체·재구축 (데이터 마이그레이션 포함).
  • 예약 정책 — 시간대별 최대수량 등 단순 정책으로 1차 구현.
  • 원장 스케줄 — 오전/오후 가능 여부만 (검안사 중심, 상세 스케줄 불필요).
  • 예약 데이터 — CRM으로 전달만, 알림톡 발송은 CRM에서 처리.
  • CRM 연동 방식 · 본인인증 업체·비용 · 카카오 자체 OAuth 전환 — 비쥬웍스가 확정·제공.

구조는 주되 원본 데이터는 주지 않는다 — 시그는 레거시 코드 재사용 아님
전달물방법개인정보
DB 스키마(DDL)mysqldump --no-data — 테이블 구조만0건
마스킹 샘플 데이터테이블당 소량(50~100행) · PII 가림/가짜값마스킹
API 명세 + mockOpenAPI/JSON Schema (비쥬웍스 제공 · 진짜 계약)—
  • 예약(마루앱) 스키마 — 재구축 핵심 참고 자료.
  • 원본 전량 이관 — 비쥬웍스 내부에서만 (시그 환경 밖 반출 금지). 레거시 소스는 필요 시 일부만 읽기용.
  • 전달 채널 — 개인 GitHub 금지 · 암호화 패키지(비번 별도) · 민감/비민감 2분리.

07꼭 챙길 것

일정·견적의 크리티컬 포인트
착수 전 / 오픈 시점 필수
  • 측정(FR-13)은 소급 불가 — 오픈 시점 미동작 시 그 기간 데이터 영구 손실. dataLayer·전환 이벤트 규격 착수 시점 합의.
  • 회원 71,580명 · 소셜 8,834명 마이그레이션 — 카카오 자체 OAuth 전환, 식별자 매핑 실패 시 로그인 불가. (비쥬웍스 선행)
  • 예약 슬롯 모델 선행 — 의료진 슬롯 규칙은 SOFTCRM 분석 후 제공. 대기 구간을 일정에 반영.
  • API 규격·mock 서버 — 착수 전 비쥬웍스 제공 (없으면 연동 대기).

08확인 필요 사항

요구사항 브리프에서 확정할 대상 — 미결 시 견적·일정 변동 요인
  • 디자인 협력사–시그 역할 범위 — 02 역할 분담은 잠정안. 디자인 협력사가 디자인까지만 산출할지, HTML/CSS 퍼블리싱(최소 JS 포함)까지 산출할지 여기서 확정.
  • 예약·인증 연동 범위 — EMR 연동 방식 · 카카오 알림톡 · 본인인증 등 연동 방식·주체 확정.
  • CMS 관리 콘텐츠 범위 확정 — 메인 배너 · 랜딩 페이지 섹션 등 어디까지 CMS로 관리할지 확정.
  • 기존 소스코드 · DB 수령 가능 여부 — 마이그레이션 편의성 관점의 수령 가능 여부 확인.
