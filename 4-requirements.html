03 · Requirements

요구사항 정리 (FR-01~16)

기능별로 As-Is → To-Be · 핵심 요구 · 수용 기준을 동일 포맷으로 정리했습니다. 수용 기준은 검수 시 그대로 시나리오로 사용합니다. 담당: 프런트·CMS = 시그 · 백엔드 API = 비쥬웍스.

우선순위 · 담당 표기 각 요구에 1순위 2순위 3순위 칩으로 우선순위를 표시했습니다. 1순위는 오픈 필수 핵심, 3순위는 후순위·선택입니다. 담당은 각 카드의 회색 줄에 명시했습니다.

A사이트 기반 구조

프런트 중심 — 백엔드는 데이터 계약만 제공
FR-01반응형 웹 1순위필수
담당 · 프런트 시그 (백엔드 의존 없음) · 제공: 모바일 스크린 13종 · URL 110개 목록
As-IsPC(/)·모바일(/m/) 별도 URL 이중 운영 — 110 URL × 2벌
→
To-Be단일 URL 반응형 · 모바일은 전용 레이아웃 별도 디자인
핵심 요구
  • 3 뷰포트(데스크톱 ≥1024 · 태블릿 768~1023 · 모바일 ≤767) 단일 코드베이스
  • 모바일 전용 레이아웃 — 단순 축소 불인정 (모바일 스크린 13종 참고)
  • 기존 /m/ URL 110개 → 301 redirect 맵 · 대표 URL(canonical) 정책
수용 기준
  • 동일 URL 뷰포트별 전환 · 모바일 시안과 구현 일치
  • redirect 맵 100% 자동 테스트 통과 · 4xx·redirect chain 0

→

  • 디자인 토큰(색·타이포·간격) 중앙 관리 · 섹션·블록 컴포넌트화
  • 템플릿 커스터마이징 범위·횟수·비용 견적 명시 (다중 템플릿은 선택 견적)

FR-03콘텐츠 데이터 연동 1순위필수

→

핵심 요구

  • 목록·상세 페이지네이션 · API 장애 시 fallback(정적 캐시)
  • 통합검색(페이지·뉴스·이벤트·블로그, 언어별 분리) — 검색 API 비쥬웍스 · UI 시그
수용 기준
  • 관리자 수정 콘텐츠 배포 없이 반영 · API 차단에도 fallback 렌더
  • 통합검색: 현 locale·발행분만 노출 · 결과 없음·페이지네이션 동작

핵심 요구

  • 비밀번호 전환(무솔트 MD5 → 최초 로그인 검증 후 BCrypt 재해시) · 휴대폰 본인인증 · 탈퇴·휴면 정책
수용 기준

  • 마이페이지에서 예약·상담·후기 조회
오픈 차단 요인 소셜 식별자 · 비밀번호 · 회원 데이터 중 하나라도 어긋나면 오픈 당일 로그인 장애 → 예약(FR-05)도 함께 막힘. 비쥬웍스 선행: 카카오 앱 소유권 확보 · 식별자 매핑.
FR-05예약 · 상담 신청 1순위필수

  • 예약 데이터 CRM 전달만 · 알림톡 CRM · 슬롯 점유 트랜잭션·멱등성·본인인증

수용 기준

  • 완료 단계 noindex · 중복 방지 · 정원 초과 방지 · 슬롯 동시성
FR-06의료진 · 진료일정 2순위필수

  • CMS 반복 일정 + 단발 예외 관리 · 휴진 등록이 예약 슬롯에 즉시 반영
  • 착수 대기: 슬롯 규칙·수용인원·변경/취소 정책은 SOFTCRM 분석 후 제공

측정은 소급 불가 오픈 시점에 태깅이 동작하지 않으면 그 기간 데이터는 영구 손실. 완료 URL·dataLayer 규격은 착수 시점에 합의해야 하며, 트래킹 자산 실측은 04 트래킹 코드 참조.
